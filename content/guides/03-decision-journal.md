Контекстно-зависимые записи архитектурных решений.

## 1. Зачем Нужен Журнал Решений

При работе с AI-ассистентами архитектурные решения принимаются быстрее, но без документирования теряется контекст. Журнал решений (Decision Log) фиксирует:

- Что было решено
- Почему именно так
- Какие альтернативы рассматривались
- Кто принял решение

## 2. Формат Записи

Каждая запись в журнале следует стандартному формату ADR (Architecture Decision Record):

```markdown
# ADR-001: Выбор базы данных

## Статус
Принято

## Контекст
Необходимо выбрать базу данных для хранения пользовательских данных
с учётом масштабирования до 100K пользователей.

## Решение
Используем PostgreSQL с SQLAlchemy ORM.

## Альтернативы
- MongoDB -- отклонено из-за сложности транзакций
- MySQL -- отклонено из-за ограничений JSON-полей

## Последствия
- Нужен опыт работы с PostgreSQL в команде
- Миграции через Alembic
- Возможность использования полнотекстового поиска
```

## 3. Организация Журнала

Рекомендуемая структура хранения:

```
docs/
└── decisions/
    ├── README.md           # Индекс решений
    ├── adr-001-database.md
    ├── adr-002-auth.md
    └── adr-003-api-style.md
```

## 4. Интеграция с AI

Журнал решений служит контекстом для AI-ассистентов:

- Ссылайтесь на ADR в `.cursorrules` или `CLAUDE.md`
- AI учитывает принятые решения при генерации кода
- Предотвращает повторное обсуждение закрытых вопросов

## 5. Лучшие Практики

- Документируйте решение сразу после принятия
- Включайте контекст и причины, а не только результат
- Ревьюйте журнал при онбординге новых участников
- Обновляйте статус при изменении решения (принято / отменено / заменено)
